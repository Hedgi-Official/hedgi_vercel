Got it — so the “quick tunnel + IPv6 bind” fixes let curl hit the endpoint instantly, **but the same request made *inside* your Express process is still stalling / timing-out.**
That almost always means the Node runtime is opening a brand-new TLS connection for every call and never re-using it, so your five-second app-level timeout fires before the handshake finishes.

---

## 1 ⃣ Drop-in fix: a single, reusable **keep-alive HTTPS agent**

```ts
// server/tunnelAgent.ts
import { Agent as HttpsAgent } from 'node:https';

/** Cloudflare tunnel: one agent per host, keep sockets open ≥ 30 s */
export const cfAgent = new HttpsAgent({
  keepAlive: true,
  keepAliveMsecs: 30_000,
  maxSockets: 10,     // enough for burst traffic
  maxFreeSockets: 5,
  timeout: 60_000     // hard kill if CF really hangs
});
```

Then, wherever you call `fetch` to the Cloudflare URL:

```ts
import fetch from 'node-fetch';
import { cfAgent } from './tunnelAgent.js';

await fetch(`${FLASK}/symbol_info?symbol=USDBRL&broker=tickmill`, {
  agent : cfAgent,     // 👈 re-uses the socket
  // you can leave your 5 000 ms timeout in place
});
```

Because the agent lives at module scope, every route handler shares the **same TLS session** and you stop paying the 300-500 ms handshake tax each time.

## 2 ⃣ Verify in 20 seconds

```bash
node - <<'JS'
  import fetch from 'node-fetch';
  import { cfAgent } from './server/tunnelAgent.js';

  const url = 'https://alleged-gb-activated-immediate.trycloudflare.com/symbol_info?symbol=USDBRL&broker=tickmill';
  const doCall = async () => {
    const t0 = Date.now();
    await fetch(url, { agent: cfAgent, timeout: 5_000 });
    return Date.now() - t0;
  };

  (async () => {
    for (let i = 0; i < 10; i++)
      console.log('call', i, 'took', await doCall(), 'ms');
  })();
JS
```

Typical output once sockets are reused:

```
call 0 took 130 ms   # first handshake
call 1 took  22 ms   # hot
call 2 took  18 ms
…
```

If the **second call** is still > 150 ms something else is breaking keep-alive (proxy, container, etc.).

## 3 ⃣ Alternative (Node ≥ 18): use the built-in `fetch` + `setGlobalDispatcher`

If you’re on Node 20 you can switch to **undici’s** global agent (already installed):

```ts
import { Agent } from 'undici';
import { setGlobalDispatcher } from 'undici';
setGlobalDispatcher(new Agent({
  keepAliveTimeout: 30_000,
  keepAliveMaxTimeout: 60_000,
}));

// now global fetch() automatically re-uses sockets
const r = await fetch(`${FLASK}/ping`);
```

That removes the need to pass an agent explicitly.

---

## 4 ⃣ Why the problem only appeared after you switched EC2 instances

* New Windows server → different ephemeral-port exhaustion / NAT rules, so sockets got closed sooner.
* Previous tunnel happened to speak **h2-mux** (single connection per origin) while the new one defaults to QUIC with a small idle timeout.
* The older Node process maybe used a long-lived agent implicitly (some frameworks do); the new build doesn’t.

Regardless, **sharing one TLS session** fixes the symptom.
