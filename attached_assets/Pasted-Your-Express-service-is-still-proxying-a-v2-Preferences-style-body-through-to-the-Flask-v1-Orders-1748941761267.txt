Your Express service is still proxying a “v2 Preferences”‐style body through to the Flask “v1 Orders” endpoint—so Flask keeps rejecting it with:

Missing or invalid v1 Orders fields:
 Required: type='online', processing_mode='automatic',
 total_amount(string), external_reference(string),
 payer(dict), transactions(dict)
In other words, you’re sending a v2 payload:

{
  type: "online",
  external_reference: "hedge_…",
  items: [ … ],
  payer: { … },
  back_urls: { … },

}
but Flask is expecting a v1 Orders‐style payload of the form:

js
Copy

{
  type: "online",
  processing_mode: "automatic",
  total_amount: "123.45",
  external_reference: "hedge_…",
  payer: { email: "…" },
  transactions: {
    payments: [
      {
        amount: "123.45",
        payment_method: { id: "...", type: "...", token: "…", installments: 1 }
      }
    ]
  }
}
You have two ways to fix this mismatch:

Option 1: Tell Express to call the v2 “createPreference” route instead of proxying to Flask’s v1 Orders
You already have a paymentService.createPreference(req, res) in Express that expects a v2 Preferences body. Just point your React/Brick code at that route. For example, in createOrder() change:

ts
Copy
Edit
// OLD — pointing at Flask’s v1 Orders, which expects a completely different schema:
const resp = await fetch("/api/payment/order", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(body),   // <— this is a v2 Preferences shape
});
to:

ts
Copy
Edit
// NEW — point at Express’s createPreference (v2 Preferences) instead:
const resp = await fetch("/api/payment/preference", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  // Same `body` as before (type, external_reference, items, payer, back_urls, auto_return)
  body: JSON.stringify(body),
});
That way, Express’s createPreference will receive exactly the fields it’s destructuring:

ts
Copy
Edit
const { type, external_reference, items, payer, back_urls, auto_return } = req.body;
and return { id, init_point, sandbox_init_point, public_key } back to React. From there you can render the v2 brick (using preferenceId: data.id) and bypass Flask’s v1 endpoint altogether.

In short:

React → call POST /api/payment/preference (not /api/payment/order).

Express handles that in paymentService.createPreference(...), creates a v2 Preference on MercadoPago, and returns { id, init_point, sandbox_init_point, public_key }.

React consumes data.id + data.public_key to build the Brick.
