The server (Express) is receiving exactly one POST to /api/payment/order that ends up returning the “mock response” (i.e. the code path under “Valid v2 Preferences payload, returning mock response”). That is why you see

Your client‐side code never actually logs 🚀 [createOrder] Function called with hedgeData, which means your React/TypeScript createOrder() function never ran at all. In other words, the only time your browser hits /api/payment/order is when you are somehow directly invoking the backend—your front‐end’s call to createOrder() never fired.

Why the Brick still appears even though createOrder() didn’t run
A previous run of renderPaymentBrick() injected the Mercado Pago iframe into <div id="paymentBrick_container">.
Once bricksBuilder.create("payment", "paymentBrick_container", settings) runs, the SDK places its own <iframe> (and all of the credit‐card form UI) inside that <div>. That iframe will remain visible until you explicitly unmount it.

You never unmounted or cleared the container after the first time you created the Brick.
That means even though your React code (with its retry guards and hasInitializedBrick flag) no longer calls createOrder(), the old iframe is still “stuck” in the DOM. Hence, when you re‐open the modal (or refresh the page without clearing that <div>), you see the Brick again, because it was never torn down.

How to fix it: Make sure you always unmount and clear before opening (or closing)
1. In your “modal close” handler, unmount and clear everything
Inside your MercadoPaySDKModal, you should have a handleClose() that is called whenever the user (or parent) wants to close the payment modal. Right at the top of that function, you must:

ts
Copy
Edit
const handleClose = () => {
  console.log("🔒 [MercadoPaySDKModal] Modal closing — tearing down any lingering Brick");

  // 1) Reset all flags so next open starts fresh:
  setPaymentCompleted(false);
  setBrickCreated(false);
  setOrderId(null);
  setPublicKey(null);

  // 2) If a Brick controller still exists, unmount it now:
  if (window.paymentBrickController) {
    try {
      window.paymentBrickController.unmount();
    } catch (e) {
      console.warn("⚠️ Error unmounting paymentBrickController:", e);
    }
    window.paymentBrickController = null;
  }

  // 3) Clear out the container’s innerHTML so no old iframe remains:
  const container = document.getElementById("paymentBrick_container");
  if (container) {
    container.innerHTML = "";
  }

  // 4) Reset the Strict‐Mode guard so that next open’s useEffect can run:
  hasInitializedBrick.current = false;

  // 5) Finally notify the parent that the modal is closed:
  onClose();
};
Why this matters: if you do not call .unmount() and set window.paymentBrickController = null, plus do container.innerHTML = "", the Mercado Pago widget remains “live” in the DOM even after your modal has been hidden. That is exactly why you still “see” the Brick even though your React code is no longer calling createOrder().

2. In your “modal open” effect, clear anything leftover before calling createOrder()
Right at the top of your useEffect (the one that listens to [isOpen, hedgeData]), insert those exact same three cleanup steps—this guarantees that each time the modal opens, you start with an empty container and no existing controller. For example:

Edit
useEffect(() => {
  console.log("🟢 [useEffect] triggered:", { isOpen, hasHedgeData: !!hedgeData });

  // 1) As soon as the modal opens, clear/unmount any leftover Brick:
  const container = document.getElementById("paymentBrick_container");
  if (container) {
    container.innerHTML = "";
  }
  if (window.paymentBrickController) {
    try {
      window.paymentBrickController.unmount();
    } catch (e) {
      console.warn("⛔️ [useEffect] Error unmounting old Brick on open:", e);
    }
    window.paymentBrickController = null;
  }
  setBrickCreated(false);
  setPaymentCompleted(false);
  setOrderId(null);
  setPublicKey(null);
  hasInitializedBrick.current = false;

  // 2) Now run your normal “if (!isOpen || !hedgeData) return;” guards:
  if (!isOpen || !hedgeData) {
    console.log("⛔️ [useEffect] Skipping initialization because modal closed or no hedgeData");
    return;
  }

    console.log("⛔️ [useEffect] Skipping because hasInitializedBrick is already true");
    return;
  }

  hasInitializedBrick.current = true;
  console.log("✅ [useEffect] Loading Mercado Pago SDK → calling createOrder()");

    .catch((err) => {
      console.error("❌ [useEffect] Failed to load Mercado Pago SDK:", err);
      setError("Failed to load payment system.");
      setLoading(false);
    });
}, [isOpen, hedgeData]);
Why this matters: if you only clear/unmount on “close” but never on “open,” there is a small gap—if the parent re-opens the modal before React’s Strict‐Mode has a chance to run handleClose, you can still end up with a leftover iframe. Clearing at the top of your “open” effect leaves no chance for a stale widget to show.

4. Review how your client calls the server
Right now your Express route for /api/payment/order does two very different things based on payload.payment_details.transactions.payments[0].payment_method.token:

If it sees a real token → It forwards everything to Flask, processes the payment, etc.

Otherwise → It interprets the request as a “create an order (v2 Preferences)” call, validates the payload, and returns:
