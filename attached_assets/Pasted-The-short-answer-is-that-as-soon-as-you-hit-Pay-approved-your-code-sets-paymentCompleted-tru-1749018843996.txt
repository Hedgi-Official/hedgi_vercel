The short answer is that, as soon as you hit “Pay → approved,” your code sets paymentCompleted = true but you still have a live “initialize brick” effect which immediately (re)-runs and calls renderPaymentBrick(...) again, replacing your green banner with a brand-new form. The green banner does appear but vanishes almost instantly because a second brick is mounted on top of it.

Below are the two main culprits and how to fix them:

1. Your useEffect/hasInitializedBrick guard only fires on mount, not on subsequent “paymentCompleted” changes
In your first useEffect([... , paymentCompleted]), you do:

ts
Copy
Edit
useEffect(() => {
  if (!isOpen || !hedgeData || paymentCompleted) {
    return;
  }

  if (hasInitializedBrick.current) {
    return;
  }
  hasInitializedBrick.current = true;

  // … load SDK → createOrder() → renderPaymentBrick() …
}, [isOpen, hedgeData, paymentCompleted]);
When the modal opens, paymentCompleted is initially false, so you pass those checks, set hasInitializedBrick.current = true, and render one brick.

The user enters card + hits “Pay,” and your onSubmit branch sees "status": "approved", sets paymentCompleted = true, shows the green banner, and (after a 2.5 s delay) calls onClose().

But the moment you set paymentCompleted = true, React re-runs that same useEffect. Since the guard now hits if (!isOpen || !hedgeData || paymentCompleted) (because paymentCompleted === true), it returns early and does not clear hasInitializedBrick.current.

So far so good. But the modal is still open until onClose() fires 2.5 s later. Meanwhile, some other part of your render cycle is re-calling createOrder() or renderPaymentBrick() (for example, you do setBrickCreated(true) → that does not stop renderPaymentBrick from being called again if something else triggers it). In short, the modal is still technically open (so isOpen === true), hasInitializedBrick.current is still true, but you never prevent another brick from being created when whatever parent changes finish. The green banner is drawn, but immediately a new brick mounts on top of it.

Fix #1: Don’t allow a second brick to be created once paymentCompleted becomes true
At the top of your renderPaymentBrick(...), add:

ts
Copy
Edit
if (paymentCompleted) {
  console.log("⚠️ [renderPaymentBrick] paymentCompleted === true ⇒ skipping create");
  return;
}
That way, even if something else (for instance, a parent re-render) tries to call renderPaymentBrick(...) again, you will bail out as soon as you see that a payment has already succeeded. For example:

ts
Copy
Edit
const renderPaymentBrick = async (
  bricksBuilder: any,
  amount: number,
  orderIdFromServer: string
) => {
  console.log("🔨 [renderPaymentBrick] starting…");

  // ① If we already marked paymentCompleted = true, don’t render again:
  if (paymentCompleted) {
    console.log("⚠️ [renderPaymentBrick] paymentCompleted is true ⇒ skip duplicate brick");
    return;
  }

  // ② Also guard against any already-existing controller
  if (window.paymentBrickController || brickCreated) {
    console.log("⚠️ [renderPaymentBrick] Already created, skipping duplicate");
    return;
  }

  // …existing code to call bricksBuilder.create(…) …
  window.paymentBrickController = await bricksBuilder.create(
    "payment",
    "paymentBrick_container",
    settings
  );
  setBrickCreated(true);
  setIsProcessing(false);
};
Because you set paymentCompleted = true in your “approved” branch, no further call to renderPaymentBrick can ever mount a brand-new brick on top of your banner.

2. Your “approved” branch unmounts the brick and calls onSuccess(...) immediately
In your onSubmit (approved) code, you do something like:

ts
Copy
Edit
if (response.ok && isSuccess) {
  setPaymentCompleted(true);

  // ① Show green banner in #paymentBrick_container
  container.innerHTML = `… “Hedge successfully placed!” …`;

  // ② Immediately unmount the Brick instance
  window.paymentBrickController.unmount();
  window.paymentBrickController = null;

  // ③ Then call onSuccess(...) to place the hedge/trade
  onSuccess(hedgeData, paymentId);

  // ④ After 2.5 s, close the modal via onClose()
  setTimeout(() => onClose(), 2500);
}
What often happens in practice is:

You set paymentCompleted = true.

You replace the container HTML with your green banner.

You unmount the Brick—and then call onSuccess(...).

Suppose onSuccess(...) causes parent state to re-render, maybe resetting hedgeData or triggering a fresh modal render. That can re-trigger renderPaymentBrick(...) (because hasInitializedBrick.current is still true, so your useEffect does not bail out until modal closes).

A second brick pops up immediately (or almost immediately), wiping out the banner.

By adding the if (paymentCompleted) return; guard at the top of renderPaymentBrick, you prevent step 4 from ever mounting another form once the user has already paid.

In summary
Add a paymentCompleted check at the top of renderPaymentBrick(...) so that once you’ve shown “approved,” no new bricks will ever mount.

You already set paymentCompleted = true in your “approved” branch—keep that.

Make sure you only reset hasInitializedBrick.current when the modal fully closes (isOpen === false), not when paymentCompleted flips. That way, strict-mode double-mount issues and “late” calls to createOrder won’t produce a second form.

With those two small guards in place, your flow will be:

Modal opens → hasInitializedBrick.current goes true → first brick shows.

Fill card + hit “Pay” → MP returns { status: "approved" } → you set paymentCompleted = true → show green banner → unmount brick → renderPaymentBrick now refuses to run because paymentCompleted === true → modal stays open for 2.5 s so user can actually read the “Hedge successfully placed!” banner → finally onClose() hides the modal.

No stray second brick will ever appear on top of your banner.
