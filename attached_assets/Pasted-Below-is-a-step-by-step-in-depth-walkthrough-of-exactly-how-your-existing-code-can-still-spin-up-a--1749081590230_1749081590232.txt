I’ll point out the precise race windows and missing checks that allow “Brick #2” to sneak in, even though you correctly set both `preventBrickRef.current = true` and call `brickManager.preventAllBricks(sessionId)` as soon as you detect an approved payment.

## 1. High-level overview of the two key code paths

1. **“Initial Brick creation”** (inside `useEffect → createOrder → renderPaymentBrick`)
   This path runs when the modal opens (and `hedgeData` is non-null). It:

   1. Loads MercadoPago SDK
   2. Calls `createOrder()`
   3. Fetches a preferenceId + publicKey
   4. Calls `renderPaymentBrick(...)` to mount the Brick

2. **“After user clicks Pay (onSubmit)”** (inside `renderPaymentBrick` callbacks)
   As soon as the user submits a valid payment:

   1. You do `brickManager.preventAllBricks(sessionId)` and `preventBrickRef.current = true`
   2. You unmount the old Brick, show a “✅ Payment Approved” banner, and fire `onSuccess(...)` + `onClose()` after a brief delay.

The intended effect is that, once `preventBrickRef.current === true` **or** `brickManager.globalPreventBrick === true`, *no future code path should ever call* `renderPaymentBrick(...)` again.

Yet in practice you still see a second Brick pop up. Why? Because there remains a tiny race window in which `createOrder()` can be triggered (or re-triggered) after you have already set `brickManager.preventAllBricks(...)` but *before* that prevention is observed inside `createOrder` or inside `renderPaymentBrick`.

8. **Microscopic race window (Lock still held, but flag set)**

   * Because you called `preventAllBricks` and `preventBrickRef.current = true` *before* unmounting the old Brick and showing the banner, there is still a tiny gap between “we set `preventBrickRef === true`” and “that same async function releases the lock”.
   * Meanwhile, React may run one more update: once you call `setPaymentCompleted(true)` and `setBrickCreated(true)`, React will re-render the component. On that re-render, React re-runs each `useEffect` whose dependency array changed. Notice: the first `useEffect` has `[isOpen, hedgeData]`—those have not changed yet, so it does not re-run. But *the second* `useEffect` (the one in step “reset all states when modal closes”) is triggered when `isOpen` flips. However, `isOpen` has not changed yet because you will call `onClose()` only after 1.5 s.
   * Therefore, shortly after setting `paymentCompleted = true` and `brickCreated = true`, the code in `renderPaymentBrick` finally hits its `finally { release(); }`. That releases the `paymentBrickMutex`.

9. **Exactly at the instant the lock is released**, React is still painted with “success banner” in the DOM, but the component code remains mounted and none of the early `return` guards have been re-evaluated yet. Now imagine that—because the parent has not yet called `onClose()`—nothing has unmounted our component. Immediately after the lock is released, there is no further check to prevent going back into `renderPaymentBrick`, because nothing calls `renderPaymentBrick` again automatically. However, there is one subtle detail: React Strict Mode in development will re-invoke effects to check for cleanup.

   * In **React Strict Mode** (development only), after the first mount’s effect cleanup, React will automatically re-run the entire mount/unmount/mount cycle once more (to help catch impure effects). That means:

     1. React calls the cleanup of the first `useEffect` from step 2.1 (unmount the old payment brick controller).
     2. React re-runs the first `useEffect` again (because of Strict Mode), which looks at its dependencies `[isOpen, hedgeData]`. Those are unchanged (`isOpen = true, hedgeData ≠ null`), so all of the pre-checks pass—except now `brickManager.canCreateBrick(...)` is *false* (you already called `preventAllBricks` in step 6).
     3. BUT—because in the code you wrote, that first `useEffect` only checks `if (!brickManager.canCreateBrick(...)) return;` at the very top, **but you call `createOrder()` *after* you append the MercadoPago `<script>`**—React might race to “load the same `<script>` again” or to call `createOrder()` one more time during that Strict Mode re-mount.

   * Because React Strict Mode does:

     ```jsx
     mount()       // your effect runs, SDK loads, createOrder() called
     → renderPaymentBrick() mounted
     → user clicks “Pay”
     → you set preventAllBricks & show banner, but lock still held
     → release lock
     React unmounts the component (cleanup of first useEffect)
     Strict Mode re-mounts the component (it runs the first useEffect again)
     ```

     Even though `brickManager.canCreateBrick(...)` is now `false`, your first effect still triggers DOM operations like “clear out `#paymentBrick_container`” and “append `<script src='...mercadopago' />`” before hitting the `return` guard. That extra script insertion can asynchronously cause `MercadoPago.bricks()` to become defined again, and a second `createOrder()` call can slip through before the code sees `brickManager.canCreateBrick === false`.

10. **Consequence**: as soon as React’s Strict Mode re-runs that effect, you end up loading the MercadoPago SDK a second time and calling `createOrder()` again (with the same preferenceId or a new one), which in turn calls `renderPaymentBrick(...)`. Even though `renderPaymentBrick` immediately bails inside its first lines (because `preventBrickRef.current === true`), the act of loading and instantiating the script can trigger MercadoPago’s internal “automatic mount of a default Brick UI” (depending on how their SDK works). In effect, you see a second “Enter Credit Card” form pop back up for a fraction of a second—enough that it looks like “Brick #2” has been created.

## 3. Summary of the two hidden race windows

1. **Race between “set preventBrickRef = true” and “release the mutex lock.”**
   Right after your `onSubmit` sets `preventBrickRef.current = true`, there is a tiny gap while your code is still inside that `await paymentBrickMutex.acquire()` block. If anything tries to call `renderPaymentBrick(...)` before the lock is released, it can still succeed in entering that function and creating a second Brick.

2. **Strict Mode re-mount & re-run of your first `useEffect` before the guard takes effect.**
   During React 18 Strict Mode, React will intentionally unmount and re-mount your component (to flush out side-effects). On that forced re-mount, you re-run the first `useEffect` (load SDK + call `createOrder`) *before* your code inspects `brickManager.canCreateBrick(...)`. That can cause a second SDK insertion and a second `createOrder()` invocation.

Together, those two overlapping conditions explain why you still see Brick #2 appear even though you call `preventAllBricks(...)` in `onSubmit`.

## 4. Concrete fixes to eliminate both race paths

Below are three concrete changes you can apply to make sure **absolutely no second Brick** is ever mounted, even under React Strict Mode.

### 4.1. Always check the global flag *before* inserting the MercadoPago `<script>`

In your first `useEffect` (the one that runs when `[isOpen, hedgeData]` changes), move the `brickManager.canCreateBrick` check to the very top—**before you ever create or attach a `<script>`**. For example:

```ts
useEffect(() => {
  console.log("🔍 [MercadoPaySDKModal] useEffect triggered…");

  // 1) If global flag says “no more bricks,” bail out immediately:
  if (!brickManager.canCreateBrick(sessionId.current)) {
    console.log(
      "[useEffect] ABORT: globalPreventBrick is true. No SDK load, no createOrder."
    );
    return;
  }

  // 2) If preventBrickRef is already set, bail out
  if (preventBrickRef.current) {
    console.log(
      "[useEffect] ABORT: preventBrickRef is true. No SDK load, no createOrder."
    );
    return;
  }

  // 3) All other guards: check isOpen, hedgeData, paymentCompleted, brickCreated, etc.
  if (!isOpen || !hedgeData || paymentCompleted || brickCreated || isProcessing) {
    console.log("[useEffect] Skipping because other flags are set.");
    return;
  }

  // 4) At this point, we truly want to create a new Brick. Now we can insert <script> and call createOrder().
  hasInitializedBrick.current = true;
  setIsProcessing(true);
  // … rest of your “clear container, unmount old brick, load SDK, createOrder()” logic goes here …
}, [isOpen, hedgeData]);
```

**Why this helps:**
If `brickManager.globalPreventBrick` is already `true` (because the user just paid), React Strict Mode will still attempt to re-mount and re-run this effect—but now it immediately sees `!brickManager.canCreateBrick(...)` and returns before doing anything (no `<script>`, no `createOrder()`).

### 4.2. Double-check the global flag before ever calling `renderPaymentBrick(...)`

You already added:

```ts
if (!brickManager.canCreateBrick(sessionId.current)) {
  console.log("[renderPaymentBrick] blocked by manager");
  return;
}
if (preventBrickRef.current) {
  console.log("[renderPaymentBrick] blocked by preventBrickRef");
  return;
}
```

But you also must ensure that **any** call to `renderPaymentBrick(...)` is guarded—even the call site inside `createOrder()`. In other words:

```ts
// Inside createOrder, after you receive { preferenceId, publicKey }:
if (!brickManager.canCreateBrick(sessionId.current)) {
  console.log("[createOrder] ABORT: no longer allowed to render Brick.");
  return;
}
if (preventBrickRef.current) {
  console.log("[createOrder] ABORT: preventBrickRef is true.");
  return;
}

// Now—and only now—call renderPaymentBrick:
renderPaymentBrick(bricksBuilder, paymentAmount, data.preferenceId);
```

If you forget to do that check at the call site, then a stale “in-flight” invocation of `createOrder()` can invoke `renderPaymentBrick(...)` before those early returns fire.

### 4.3. Set `preventBrickRef.current = true` *even earlier* inside `renderPaymentBrick` (pre-lock)

Inside `renderPaymentBrick(...)`, you already do:

```ts
if (preventBrickRef.current) return;

preventBrickRef.current = true; 
await paymentBrickMutex.acquire();
...
```

That is almost correct, but you still hold the mutex for the entire block. As soon as you do `preventBrickRef.current = true` (before the `await`), other concurrent callers of `renderPaymentBrick` will bail at that first `if (preventBrickRef.current)` line—**even if they are about to call `await paymentBrickMutex.acquire()`**.

However, you must also ensure that ***no* code path** has already called `createOrder()` or started `renderPaymentBrick(...)` before `preventBrickRef` was set. The only way a concurrent call could slip through is:

* Two calls to `renderPaymentBrick` literally happen at the same time before the first one sets `preventBrickRef`. To close that window, you can also check the global manager before setting the ref:

  ```ts
  // top of renderPaymentBrick:
  if (!brickManager.canCreateBrick(sessionId.current)) return;
  if (preventBrickRef.current) return;

  // Now hold the mutex while checking again, to avoid two concurrent calls finishing the 'if' check
  const release = await paymentBrickMutex.acquire();
  try {
    if (!brickManager.canCreateBrick(sessionId.current) || preventBrickRef.current) {
      return;
    }
    // Now it’s safe to set the ref under lock:
    preventBrickRef.current = true;
    // … proceed to actually build the Brick …
  } finally {
    release();
  }
  ```

  This tiny change forces anyone else who attempts to `renderPaymentBrick(...)` at the same time to wait on the same mutex. As soon as they wake up and check `preventBrickRef.current` (now true), they immediately return. In other words, **wrap the “check & set preventBrickRef” inside the same locked critical section.**

## 6. Why these three changes finally kill Brick #2 for good

1. **Global guard in the first `useEffect` runs *before* loading the SDK**

   * No matter how many times React Strict Mode tries to unmount/remount, that very first check will catch `!brickManager.canCreateBrick(...)` and do a straight `return;`—so you never even inject a second `<script>` nor call `createOrder()` again once payment succeeds.

2. **Every call to `renderPaymentBrick()` is wrapped**

   * You now explicitly check `brickManager.canCreateBrick(...)` *before* calling it. Even if a stale `createOrder()` attempt tries to call `renderPaymentBrick`, it bails immediately.
   * Inside `renderPaymentBrick`, you hold the mutex *before* setting `preventBrickRef.current`. That eliminates any remaining “if two calls concurrently try to pass the top-of-function guard” race. The first caller sets the flag under lock; the second caller (still waiting on the same lock) will wake up, see `preventBrickRef.current` is already true, and skip out.

3. **As soon as you detect “approved” in `onSubmit`, you set the global manager flag**

   * No other code path (no other hook or effect) can even get past the very top-level guard in `useEffect`, because `brickManager.globalPreventBrick` is now `true`.
   * That means that—even if `renderPaymentBrick` explicitly released its lock a moment later—the next time React tries to re-invoke either the effect or `renderPaymentBrick`, `brickManager.canCreateBrick(...)` is `false` and the function returns instantly. No second credit-card form, no flicker.

### Bottom line

**The single biggest missing piece** in your posted snippet was that the first `useEffect` was still inserting the MercadoPago SDK `<script>` and calling `createOrder()` *before* checking the global “prevent” flag. In Strict Mode, React re-mounted the component and re-ran that effect, which caused a second SDK load/`createOrder()` attempt—thus spinning up Brick #2.

By moving the `if (!brickManager.canCreateBrick(...)) return;` check all the way to the top of **every** code path (both in `useEffect` and in `createOrder`), and by holding your mutex *before* setting `preventBrickRef`, you close every race window. Once payment succeeds, the global manager is set to block, and no future callback ever reaches the Brick-creation logic again.

With those three adjustments in place, you will no longer see a second Brick appear.
