Trading Functionality Debugging Analysis
After reviewing the codebase thoroughly, I can identify several potential issues that might be causing the trading functionality to behave incorrectly. Let's analyze what could be happening when you place a USDMXN buy trade but end up with a EURUSD sell trade.

Potential Issues
1. Server Connection Timeouts
The logs show multiple connection timeout errors to the XTB server at http://3.147.6.168:5000:

[Trading Service] XTB server health check failed: FetchError: request to http://3.147.6.168:5000/ failed, reason: connect ETIMEDOUT 3.147.6.168:5000
This suggests that the server is either:

Not responding at all
The IP/port combination might be incorrect
Firewall issues preventing connection
When connection fails, the system falls back to simulation mode, which might explain discrepancies.

2. Parameter Formatting Issues
In trading.ts, I notice potential issues with how trade parameters are formatted:

Volume Conversion: The system converts volume from units to lots by dividing by 100,000:

const adjustedVolume = volume / 100000;
If this conversion is incorrect for the XTB API expected format, it could cause issues.

Trade Command Structure: In server/services/trading.ts, the trade transaction info structure might not match what the API expects:

const tradeTransInfo: any = {
  cmd: isBuy ? 0 : 1,     // 0 for BUY, 1 for SELL
  symbol: symbol,
  volume: finalVolume,
  price: price || 0,
  offset: 0,
  order: 0
};
3. Command/Type Confusion
There's inconsistency between different implementation files:

In XTBTrader.py:

trade_info = {
  "cmd": TransactionSide.BUY if is_buy else TransactionSide.SELL,
  "symbol": symbol,
  "volume": volume,
  "type": TransactionType.ORDER_OPEN,
  "price": 0.0
}
In trading.ts [simulation mode]:

const simulatedOrder = Math.floor(1000000 + Math.random() * 9000000);
In xtb_bridge.py, a different parameter structure is used:

trade_info = {
  "cmd": request.command,
  "symbol": request.symbol,
  "volume": request.volume,
  "type": request.orderType,
  "price": getattr(request, 'price', 0.0),
  ...
}
4. Fallback/Simulation Mode Behavior
When the server is unreachable, the system falls into a simulation mode that might be using hardcoded values or incorrect behavior:

} catch (statusError) {
  console.warn(`[Trading Service] Failed to check trade status with server, using simulated data for trade ${tradeNumber}:`, statusError);
  // ... simulated data return ...
}
5. Authentication Issues
The login process might not be working correctly, causing subsequent trade commands to fail or be misinterpreted:

const userId = Number(process.env.XTB_USER_ID || '17535100');
If the authentication is failing but simulation mode is hiding this, trades won't be properly executed.

6. Two Different XTB Client Implementations
You have both:

The TypeScript implementation in trading.ts
Python implementations in XTBTrader.py and server/services/XTBTrader.py
They might be interfering with each other or using different authentication sessions.

What Happens When A Trade Is Placed
When you place a trade:

The frontend sends a request to your Node.js backend
The backend tries to connect to the XTB server (currently timing out)
After multiple failed retries, it falls back to simulation mode
In simulation mode, it returns what looks like a successful response but doesn't actually place a trade
When the actual server becomes available again, it might be processing backed-up commands or using cached values
