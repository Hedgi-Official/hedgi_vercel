The reliable way to guarantee “never create a second brick” is to introduce a mutable ref that you flip to true as soon as you know “payment succeeded.” Because a ref update is instantaneous (not subject to React’s render batching), you can check that ref at the very top of your effect or brick-render function and bail out before any brick logic runs.

Create a ref:

ts
Copy
Edit
const preventBrickRef = useRef(false);
In your success path (inside onSubmit), immediately do:

ts
Copy
Edit
preventBrickRef.current = true;
setPaymentCompleted(true);
setBrickCreated(true);
// unmount the old brick, inject your banner, etc.
As soon as you set preventBrickRef.current = true, any further calls to your effect or renderPaymentBrick will “see” that flag and return immediately.

Guard your effect and your brick-render function:

ts
Copy
Edit
useEffect(() => {
  if (
    !isOpen ||
    !hedgeData ||
    paymentCompleted ||
    brickCreated ||
    preventBrickRef.current
  ) {
    return;
  }
  // … load SDK and call renderPaymentBrick() here …
}, [isOpen, hedgeData, paymentCompleted, brickCreated]);

const renderPaymentBrick = async (publicKey, preferenceId) => {
  if (preventBrickRef.current) {
    return; // bail out right away
  }
  // …normal logic to create payment brick…
};
Because preventBrickRef.current is set to true synchronously (not queued like state), there is no “bump in the road” where React still thinks the flags are false. The moment you assign preventBrickRef.current = true, your next effect run or brick-render call will “see” that and immediately exit. That ensures the credit card form never re-initializes once payment has succeeded.

TL;DR
State updates are batched: calling setPaymentCompleted(true) doesn’t flip paymentCompleted until after your handler finishes.

In that tiny gap, your effect can re-run and see “Both flags are still false,” so it re-creates the brick.

Solution: use a mutable ref (preventBrickRef.current = true) the instant you know “payment approved.” Check that ref at the very top of any brick-creation logic, and bail out if it’s set. That ref update is immediate and closes the window for any second brick to appear.
